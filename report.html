1. Introduction

Seminar Track is a full-stack web application developed as a course project for the Web Programming Languages (WPL) module. The system is designed to digitise and streamline the management of university seminars — replacing manual, paper-based processes with an efficient, role-based web platform accessible from any modern browser.

The application caters to three distinct user groups: Administrators, who control the entire platform; Teachers, who are assigned to seminars and manage attendance; and Students / Public Users, who browse upcoming seminars and self-register without requiring an account.

The project demonstrates practical application of core web programming concepts including server-side scripting with PHP, relational database management with MySQL, responsive UI design with Bootstrap 5, and web security best practices such as CSRF protection and password hashing.

2. Project Objectives

The primary objectives of the Seminar Track system are:

  1. Provide a centralised platform for creating, managing, and publishing university seminars.
  2. Allow public self-registration for seminars without requiring students to create accounts.
  3. Enable role-based access control separating Admin, Teacher, and public user capabilities.
  4. Automate capacity tracking to prevent over-registration at seminars.
  5. Facilitate digital attendance marking by assigned teachers during live seminars.
  6. Generate attendance reports per seminar for administrative review.
  7. Demonstrate proficiency in Core PHP, PDO, MySQL, HTML/CSS, Bootstrap 5, and secure web development practices.

4. System Architecture

The application follows a three-tier architecture without using an MVC framework, keeping the codebase straightforward and aligned with Core PHP learning objectives.

4.1 Presentation Tier

All HTML output is generated server-side using PHP templates. Shared layout components (public_header.php, public_footer.php, admin_header.php, teacher_header.php) are included in every page via require_once, ensuring consistent navigation, CSS, and JavaScript across the entire application.

4.2 Logic / Application Tier

Business logic is organised into reusable function libraries inside the includes/ directory. The key files are:

  • config.php — site-wide constants (DB credentials, site name, base URL)
  • db.php — singleton PDO connection factory (getDB())
  • functions.php — all query functions, helpers (formatting, flash messages, CSRF)
  • auth.php — session management, login, logout, role guards

4.3 Data Tier

MySQL 8 with InnoDB storage engine. All tables use utf8mb4 character set to support full Unicode. Relationships are enforced via foreign key constraints with appropriate ON DELETE CASCADE / SET NULL rules.

Design Decision: A singleton pattern is used for the database connection (static $pdo = null inside getDB()), ensuring only one PDO instance is created per request regardless of how many times getDB() is called.

5. Database Design

5.1 Entity-Relationship Overview

The database (seminar_db) contains five tables with the following relationships:

  • A User can have role admin or teacher.
  • A Seminar is created by a User (admin) and can be assigned to many Teachers (via seminar_teachers).
  • A Registration belongs to one Seminar. Students register publicly without a user account.
  • An Attendance record is created per Registration per Seminar, marked by a Teacher (User).

7. Security Implementation

Security is implemented at multiple layers following OWASP guidelines relevant to this scale of PHP application.

7.1 SQL Injection Prevention

Every database query uses PDO prepared statements with parameter binding. User-supplied values are never interpolated directly into SQL strings. 

// Safe — parameterised query
$stmt = $db->prepare('SELECT * FROM users WHERE email = ? AND role = ? LIMIT 1');
$stmt->execute([$email, $role]);

// Never done — vulnerable interpolation
$db->query("SELECT * FROM users WHERE email = '$email'");  // ✗ NOT used

7.2 Cross-Site Scripting (XSS) Prevention

All output rendered into HTML passes through the e() helper function, which calls htmlspecialchars() with ENT_QUOTES | ENT_SUBSTITUTE. This neutralises any injected HTML or JavaScript from user input. 

function e(string $s): string {
    return htmlspecialchars($s, ENT_QUOTES | ENT_SUBSTITUTE, 'UTF-8');
}

// Usage in templates
echo e($student['name']);  // <script> becomes harmless text

7.3 CSRF Protection

All state-changing forms include a hidden CSRF token generated by csrfToken() and validated by verifyCsrf() before any POST handler executes. The token is stored in the user's PHP session and compared using timing-safe hash_equals(). 

// In every form
<input type="hidden" name="csrf_token" value="<?= csrfToken() ?>">

// At the top of every POST handler
verifyCsrf();  // Calls hash_equals() — kills the request on mismatch (HTTP 403)

7.4 Password Hashing

Passwords are stored exclusively as bcrypt hashes using PHP's native password_hash() with PASSWORD_DEFAULT. Verification uses password_verify(), which is resistant to timing attacks. 

// Storing a password
password_hash($plaintext, PASSWORD_DEFAULT)  // Produces $2y$10$... bcrypt hash

// Verifying at login
password_verify($submitted, $stored_hash)    // Timing-safe comparison

7.5 Session Security

Upon successful login, session_regenerate_id(true) is called to issue a new session ID, preventing session fixation attacks. On logout, the session array is cleared, the cookie is expired, and session_destroy() is called.

7.6 Role-Based Access Control

Every admin page begins with requireAdmin() and every teacher page with requireTeacher(). Both functions check the session role and immediately redirect unauthorised users to the appropriate login page.

8. Key PHP Concepts Demonstrated

8.1 PDO and Prepared Statements

The getDB() function creates a single PDO instance configured with ERRMODE_EXCEPTION, FETCH_ASSOC, and EMULATE_PREPARES = false (uses true server-side prepared statements). All CRUD operations use prepare() / execute() with bound parameters.

8.2 Session Management

PHP sessions store authenticated user state (user_id, user_name, role) between HTTP requests. Flash messages for success/error feedback are also stored in the session and consumed once via getFlash()/setFlash().

8.3 Singleton Design Pattern

The getDB() database connection factory uses PHP's static local variable to implement the Singleton pattern — ensuring only one database connection exists per request, regardless of how many files call getDB(). 

function getDB(): PDO {
    static $pdo = null;          // Persists across calls within same request
    if ($pdo === null) {
        $pdo = new PDO(...);     // Created only once
    }
    return $pdo;
}

8.4 AJAX with Fetch API

The attendance marking feature uses the browser's native fetch() API to send asynchronous POST requests to ajax/mark_attendance.php. The PHP endpoint returns a JSON response (Content-Type: application/json) which JavaScript parses to update the UI without a page reload.

8.5 Form Validation (Server + Client)

All forms perform dual validation: client-side using jQuery Validate (immediate feedback) and server-side in PHP (authoritative check). Server-side validation is always the final gatekeeper, since client-side can be bypassed.

8.6 Include / Require Architecture

Rather than duplicating HTML, every page require_onces the appropriate header/footer file. The header files accept a $pageTitle variable set before inclusion to customise each page's <title> tag dynamically.

8.7 PHP Closures and Arrow Functions (PHP 7.4+)

Array filtering in the Teacher dashboard uses PHP 8 arrow function syntax: 

$upcoming = array_filter($mySeminars, fn($s) => $s['status'] === 'upcoming');

8.8 Environment-Aware Configuration

config.php reads database credentials from environment variables first (getenv()), falling back to hardcoded development defaults. This allows the same codebase to run in development (Laragon) and production (server with env vars set) without code changes. 

define('DB_HOST', getenv('DB_HOST') ?: 'localhost');
define('DB_PASS', getenv('DB_PASS') ?: '');

11. Challenges & Solutions

ChallengeSolution
Capacity Race Condition
Two students submitting simultaneously could both pass the "seats available" check.		 Capacity is re-checked server-side inside the POST handler after CSRF validation. The check and insert happen within the same request, and MySQL's row-level locking on the registrations table prevents double-booking at the database level.
Duplicate Registration
A student could register multiple times with the same email.		 The isAlreadyRegistered() function queries for an existing (seminar_id, student_email) pair before inserting. An error message is shown if a duplicate is detected.
Attendance AJAX CSRF
AJAX requests need CSRF tokens but don't use regular form submission.		 The CSRF token is embedded in a <meta> tag in the page head. JavaScript reads it via document.querySelector('meta[name=csrf-token]') and includes it in every fetch() POST body.
Teacher Data Isolation
A logged-in teacher should not see other teachers' seminars.		 All teacher queries use JOIN seminar_teachers ON teacher_id = ? with the session's user_id as the parameter. Teachers cannot access seminar data beyond their assignments at the database query level.
Shared Stylesheet
Three separate portals (public, admin, teacher) need distinct but consistent styling.		 A single style.css file uses scoped CSS classes (.admin-body, .teacher-body, .sidebar-admin, .sidebar-teacher) to differentiate portal themes without duplicating common rules.
Environment Portability
Hard-coded credentials break when deploying from development to production.		 config.php reads all connection settings from environment variables (getenv()) and falls back to development defaults only if the variable is not set, making the application deployment-ready without code modifications.

12. Conclusion

Seminar Track successfully demonstrates the practical application of full-stack web development using Core PHP and MySQL. The project covers the complete software development cycle — from database schema design and server-side business logic through to responsive user interface construction and deployment scripting.

Key learning outcomes achieved through this project include:

  • Writing secure, parameterised SQL queries using PDO to prevent injection attacks
  • Implementing role-based access control entirely in server-side PHP without frameworks
  • Designing a normalised relational database with foreign key integrity constraints
  • Building asynchronous user interfaces with the Fetch API and JSON-returning PHP endpoints
  • Applying OWASP security practices: CSRF tokens, XSS output encoding, and bcrypt password storage
  • Organising a multi-file PHP project using reusable includes, a singleton DB connection, and helper function libraries

The system is functional, tested locally on Laragon, and ready for demonstration. Potential future enhancements include email confirmation on registration, PDF certificate generation for attended seminars, and export of attendance reports to Excel.
